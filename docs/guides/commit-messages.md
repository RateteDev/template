# コミットルール

通常コミットとマージコミットの両方に適用する。件名フォーマットと本文構成のみ種別ごとに異なり、その他のルールは共通とする。

## 件名フォーマット

### 通常コミット

```
[{scope}]{type}: {日本語の概要}
```

- `{scope}`: 変更対象範囲を表現する。モノレポ構成を前提とする。
    - レポジトリ全体にまたがる場合には[repo]、特定のアプリケーション内に閉じた変更であればそれを使用する。例: [frontend], [android]
- `{type}`: `feat` / `fix` / `refactor` / `docs` / `chore` のいずれかを使用する。これら以外は使用しない

### マージコミット

```
{branch} をマージ: {日本語の概要}
```

- `{branch}`: マージ元のブランチ名

いずれの種別も `{日本語の概要}` は体言止めで記述し、文末に句点は付けない。

## 文体のルール

- 背景は無機質な箇条書きで済ませず、散文で読みやすく説明する
- 内部仕様の詳細（ファイルパス、関数名、アルゴリズムの実装手順など）の列挙ではなく、外部から観測できる振る舞いや呼び出し方法の変化を中心に記述する
- 散文部では英単語と日本語の混在を最小限にする。識別子・コマンド名・固有名詞は原語のまま残し、一般語は日本語へ置き換える（例: ×「attach させる」 ◯「接続させる」）
- すべての変更を網羅しようとせず、重要な変更のみに焦点を絞り簡潔にまとめる
- 細部は差分から読み取れるため、本文では多少抽象化してでもアーキテクチャの変更点や処理フローの全体像を読み手に伝えることを優先する
- 非公開（private）の別リポジトリは、読者が参照を確認できず情報として意味を持たないため言及しない。OSSなど公開されている参照先であれば言及してよい

## 作成ルール

- 本文は必ず記述
- 500文字以内に収める（+20%程度なら許容）。必ず1回は`wc -m {path}` で計測を行う
- 文字数超過の際は安易に単語の言い換えや修飾語の削除で対処せず、重要度の高い情報に絞ることや抽象化した表現で対処する
- 完成したコミットメッセージはドラフトとしてユーザー指定の一時ディレクトリに`commit-draft-{commit title}.md`の形式で保存する
- コミット後ドラフトファイルを削除する

## 確認フロー

- 本文冒頭節（通常コミットの`## 背景`、マージコミットの`## サマリー`）の内容が、会話でユーザーが明言した理由や差分から機械的に読み取れる事実のみで構成できる場合、確認を取らずそのまま`-F`オプションでコミットまで完了させる
- 意図・目的・経緯についてユーザーが明言していない部分を推測で補って記述した場合は、コミット前に要点を対話上で簡潔に提示し、確認を取ってから進める。ドラフト全文やファイルを開いての確認は求めない
- 明確か推測かの判断に迷う場合は、確認を取る側に倒す

## 本文構成

本文は種別に応じた以下の3節を、この順序で必ず配置する。

### 通常コミット

1. `## 背景`
2. `## 変更サマリー`
3. `## 効果・影響`

### マージコミット

1. `## サマリー`
2. `## 含まれる変更`
3. `## 残課題`

## 例

### 通常コミット

```markdown
[backend]feat: 投稿検索エンドポイントへのタグ絞り込み追加

## 背景
検索画面は複数タグ指定に既に対応済みだったが、バックエンド側エンドポイント`GET /api/posts`はフリーワード検索しか受け付けていない状況だったため、API側の対応が必要だった

## 変更サマリー
- `GET /api/posts` に `tags` パラメータを追加（CSV、最大10件）
- 空値・重複・未知タグのバリデーションを追加

## 効果・影響
- この変更によってクライアント側はタグ絞り込み機能を実装できるようになった
- `tags`パラメータ省略時の挙動は変更なしで後方互換を維持
```

### マージコミット

```markdown
prep/url-preview-fixtures をマージ: URL抽出テスト用 fixture を統一し、プレビュー CLI を追加

## サマリー
URLプレビュー実装に向けて、テスト用データ（fixture）を仕様 v1.1.1 に統一し、`data/fixtures/url-extraction.json` に集約した。
同時にプレビュー確認用の CLI を `tools/preview-url-extraction.sh` に追加した。
今後は API / バックエンド / Flutter がこのデータを共通の期待値として参照する。

## 含まれる変更
- エラー分類を整理（`malformed:host-too-short` を廃止、`malformed:host-unreachable` に統一）
- Markdown 形式（`[text](url)`）は対象外と明記
- IDN 変換失敗ケースを追加（テストケース計 68 種）
- 結果を色分け表示（緑=有効 / 黄=除外 / 赤=エラー / 灰=URL なし）
- 環境変数で結果を絞り込み可能（`URL_PREVIEW_FILTER` / `URL_PREVIEW_LIMIT`）

## 残課題
- バックエンド・Flutter 側の参照実装は後続 Phase で対応
```
