# テストガイド

テストの配置と実行方針を定める。テストランナーの選定と各分類の責務の確定はプロジェクト側の裁量とし、本ガイドはランナー非依存の構造と原則のみを規定する。

## ディレクトリ構造

テストは各パッケージ内の `tests/` 配下に置き、トップレベルは次の 4 分類で固定する。

| 分類 | 検証の主眼 | 目安 |
|---|---|---|
| `unit` | 単一の公開インターフェースの振る舞い | 外部 I/O を持たない。内部で呼ぶ自作モジュール・ライブラリは差し替えず実物のまま用いる。その信頼は各モジュール自身のテストが担保する |
| `integration` | モジュール間の連携そのもの、または実 I/O との接続 | 開発者視点で内部の結合点を突く。連携が主眼なら I/O がなくても integration に置き、実 I/O に触れる検証はすべてここ以上に置く |
| `e2e` | 利用者視点のフロー全体 | 内部構造を知らない立場で、公開された操作面（CLI・API・UI）だけを使いシステムを外から操作する |

`utils` はテスト専用のヘルパー・前提検証ユーティリティの置き場であり、テストの分類ではない。

### 分類の判定

迷った場合は上から順に判定する。

1. システムの外から利用者として操作しているか — はい: `e2e`
2. 実 I/O に触れるか、複数モジュールの連携そのものを検証したいか — はい: `integration`
3. それ以外（単一の公開入口の振る舞いで I/O なし） — `unit`

例:

- 複数のヘルパーを内部で呼ぶ純粋なパース関数を、公開入口から検証する — `unit`。合成でも主眼は単一入口である
- リポジトリ層が実 DB に正しく読み書きできるか — `integration`
- サービス層とリポジトリ層のトランザクション境界の協調 — `integration`。I/O の有無によらず連携が主眼である
- ビルドした CLI をコマンドとして実行し、生成物が実際に機能するか検証する — `e2e`

integration と e2e の境界は視点で切る。同じリクエストを投げるテストでも、内部の結合点を開発者の知識で突くなら integration、公開された操作面だけで利用者のシナリオをなぞるなら e2e とする。

### 内部構造

各分類の内部はサブディレクトリで切る。目安は次のとおり。

- `unit` — `src/` の構造をミラーする。テストの所在が機械的に定まり、対応する実装が消えた孤児テストを検出できる
- `integration` — 結合点単位で切る。結合テストは複数モジュールにまたがるため、`src/` ミラーは適用しない
- `e2e` — スイート単位で切る。対象や前提が異なる検証は別スイートに分ける（例: CLI ツールにおける `e2e/cli/` = CLI の操作、`e2e/output/` = 生成物に対する実操作）

```
apps/{app}/tests/
├── README.md
├── unit/
├── integration/
│   └── {結合点名}/
├── e2e/
│   ├── {スイート名}/
│   └── {スイート名}/
└── utils/
```

### tests/README.md

各パッケージの `tests/README.md` に、そのプロジェクトにおける各分類の責務と内部の切り方を定義し、以後の運用はその定義に従う。後述する例外（記録再生プロキシ）を採用した場合の記録もここに置く。

### 実行レシピ

テストの実行は just レシピを通す。前提が異なるスイートには対応するレシピを用意し、実行単位の独立性はディレクトリ構造ではなくレシピで表明する。

## 実呼び出しの原則

テストコードでのモック使用を一切禁止する。テストコードは本番同等の経路のまま、実 API・実コマンドを呼び出す。モックはテストに「実装と別の世界」を見せ、テストは緑なのに本番で壊れる乖離と、モック自体の管理責任を生むためである。

### 前提検証

実呼び出しには環境の充足が前提となる。`beforeAll` 等で前提（環境変数・向き先の到達性・依存サービスの状態）を検証し、未達の場合は skip ではなく fail として早期に落とす。skip は環境不備を緑に見せかけ、CI に嘘を持ち込む。

検証には、向き先が既知の本番アドレスと一致しないことを含める。書き込みや破壊的操作が設定ミスで本番へ到達する事故を実行時に止める、最後の防壁である。

### 書き込みを伴うテスト

書き込み系の実呼び出しは、`beforeAll` 等で必要な状態を準備し、`afterAll` や各テスト後のクリーンアップで元の状態に戻す。

- テストが作成する成果物には識別可能な接頭辞と実行 ID を付与する（例: `test-{実行ID}-{名前}`）。クリーンアップが走らなかった場合の残骸を後から特定・掃除できるようにするためである
- 書き込みを伴うテストを実装する際は、事前にユーザーへの確認を必須とする

## 向き先の切り替え

テスト用インスタンスやサンドボックスへ向き先を変える場合も、テストコードは無改変とし、切り替えは just レシピで行う。

- 本番相当の実行とモックへ曲げた実行は別レシピに分ける（例: `just test` / `just test-mock`）。どの向き先で実行したかが、コマンドの字面に常に現れるようにする
- モック側のレシピは、プロキシの起動 → 向き先の設定 → テスト実行 → 停止までを内部で一括して行う。向き先を決める機構（環境変数・設定ファイル等）はレシピ内部に隠蔽し、利用者とテストコードには見せない
- レシピの外に永続する状態を残さない。有効化レシピと実行レシピを分けるトグル型は、残留状態が次回以降の実行を黙って曲げるため採用しない
- 向き先が解決できない場合はフォールバックせず fail とする

## 例外: 記録再生プロキシ

実呼び出しが次のいずれかに該当する場合に限り、最終手段として記録再生プロキシ（実 API の応答を記録し再生する固定レスポンス）を許容する。

- 課金が発生する
- レート制限に抵触する
- 破壊的操作を伴う
- DoS 相当の負荷を与える

優先順位は 実物 → サンドボックス・テスト用インスタンス → 記録再生プロキシ とし、上位で成立するなら下位を選ばない。

採用する場合は次を必須とする。

- 固定レスポンスの手書きを禁止し、実 API を一度呼び出して応答を記録・再現する（record & replay）。手書きは開発者の思い込みの写しであり、モックの害がそのまま残る
- 例外の存在と事由を `tests/README.md` に記録する
- 記録日と記録方法をモック実装のソースコード冒頭に記載する
- 記録物はコミット前にサニタイズし、トークン・個人情報等の機微情報を除去する。手動確認に加えて機械的なシークレットスキャンを併用し、ツールの選定はプロジェクト側で行う
