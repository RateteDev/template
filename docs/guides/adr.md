# ADR ガイド

プロジェクトの重要な設計判断を ADR (Architecture Decision Record) として記録する。

## 目的

「何を選んだか」だけでなく「なぜそれを選んだか」を後から追跡できるようにする。採用した判断だけでなく、検討の末に見送った判断も記録する。

## 運用

- ADR は `docs/adr/` 配下に 1 判断 1 ファイルで置く。
- ファイル名は `YYYYMMDD-kebab-case-title.md` とする。日付は判断を記録した日を表し、ディレクトリ内の時系列ソートを担う。
- 一度書いた ADR は書き換えない。判断が覆った場合は新しい ADR を追加して積み重ねる。状態管理のメタデータは持たず、日付の新しい ADR が古い ADR に優先する。
- ADR ファイルを追加するコミットの件名末尾には `#ADR` タグを付け、`git log --grep='#ADR'` で判断の履歴を追えるようにする。

## フォーマット

`背景` と `判断` は必須とし、他のセクションは該当する場合のみ書く。

構成・フロー・案の分岐など、文章より図が伝わる箇所では mermaid を積極的に使う。特に背景での現状説明に有効である。

````markdown
# {判断のタイトル}

## 背景

{何が問題で、どんな制約・経緯があったか}

## 判断

{何を採用 / 見送りしたか}

理由:
- {箇条書きで}

## 検証

{判断の裏付けとなる実験・計測}

## 影響

{この判断が他へ波及する範囲}

## 再検討の条件

{どうなったらこの判断を見直すか}
````
