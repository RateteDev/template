# ADR ガイド

プロジェクトの重要な設計判断を ADR (Architecture Decision Record) として記録する。

## 目的

「何を選んだか」だけでなく「なぜそれを選んだか」を後から追跡できるようにする。採用した判断だけでなく、検討の末に見送った判断や、判断材料となる調査・検証の結果も記録する。

## 運用

- ADR は `docs/adr/` 配下に 1 判断 1 ファイルで置く。
- ファイル名は `YYYYMMDD-hhmm-{h1のタイトル}.md` とする。日時は判断を記録した日時を表し、ディレクトリ内の時系列ソートと同日複数 ADR の順序付けを担う。
- タイトル部分は h1 と完全一致させる。一致を機械的に保てるよう、h1 にはバッククォート・空白・`/` を含めない。
- 一度書いた ADR は書き換えない。判断が覆った場合は新しい ADR を追加して積み重ねる。状態管理のメタデータは持たず、日時の新しい ADR が古い ADR に優先する。
- 過去の ADR に触れる場合はファイル名で参照し、背景の書き出しで経緯を接続する。
- ADR ファイルを追加するコミットの件名末尾には `#ADR` タグを付け、`git log --grep='#ADR'` で判断の履歴を追えるようにする。

## フォーマット

文書は h1 のタイトル 1 つで始める。h2 は `背景`・`判断`・`根拠` の 3 つに固定し、この順で書く。h2 は増やさない。内容の多様性は各セクション配下の h3 で表現する。

本文に日付や状態を書かない。日時はファイル名とコミットが、判断の新旧は ADR の積み重ねが担う。

- `背景` — 何が問題で、どんな制約・経緯からこの検討に至ったか。
- `判断` — 何を採用 / 見送り / 保留したか。結論を 1〜2 文で先に述べる。確定していない場合も「何を決めて何を決めていないか」を明示する。
- `根拠` — 判断を裏付ける理由・検証・調査の詳細。

h3 の例:

- `判断` 配下: `影響`（判断そのものの条件ではなく、判断の外側へ波及する変化）/ `未決事項`（本 ADR で扱わず後続の判断に残した論点。次の ADR の背景がここに接続する）
- `根拠` 配下: `理由`（箇条書き）/ `検証結果`（実験・計測）/ `調査結果`（外部仕様・代替案の比較）

`理由` には判断の決め手だけを書く。背景で述べた問題の再説明や、検証結果・調査結果に載せる数値の再掲は理由に含めない。

テンプレート:

````markdown
# {判断のタイトル}

## 背景

{問題・制約・ここに至る経緯}

## 判断

{採用 / 見送り / 保留の結論を 1〜2 文で}

### 影響

{該当する場合のみ}

## 根拠

### 理由

- {箇条書き}

### 検証結果

{該当する場合のみ}
````

## 書き方

- 常体（〜である・〜とした）で書く。テンプレートの穴埋めではなく、経緯がそのまま読める文章にする。
- 背景は 2〜4 文で「問題 → 試した経緯 → なぜこの検討に至ったか」を圧縮する。
- 具体値を書く。バージョン・HTTP ステータス・計測値・定数名など、判断の材料になった数字と名前を残す。「問題があった」のような抽象表現だけで済ませない。
- ファイル名・関数名・コマンド名・ライブラリ名などのコード識別子は必ずインラインコード（`` ` `` で囲む）にする。ただし h1 はファイル名と一致させるため例外とし、識別子も装飾なしで書く。
- 地の文は短文を並べない。「〜する。〜する。」ではなく「〜に変更し、既存の〜は廃止する」のように接続する。ただし 1 文は 100 字を目安とし、超えるなら文を分ける。
- 箇条書きは逆に切り詰める。1 項目 80 字以内・読点 2 つまでを目安とし、体言止めを許容する。収まらない内容は箇条書きをやめ、**強調** を見出し代わりに置いた文章か h3 へ格上げする。
- 同じ事実を繰り返さない。背景で述べた失敗や制約を根拠で再度書く場合は、表への整理など別の形に寄せる。
- 「〜する予定である」といった将来の約束は書かない。未決の論点は `判断` 配下の `未決事項` に書く。

## 図表の使い分け

- 比較・一覧は表に、注意・制約・リスクは Callout に、構造は mermaid に寄せる。
- どの h2・h3 でも「一目で概要を掴めるか」を基準に組み立てる。関係・流れ・収束といった構造を持つ内容は、文章で説明する前に mermaid で示し、文章は図の補足に回す。
- よく使う型:
    - **収束図** — 複数の理由・条件が 1 つの判断へ矢印で集約する `flowchart`。「ビルドが高速」「エッジで動く」「開発が盛ん」→「◯◯を採用」のように、判断の全体像を一目で示せる。
    - **処理フロー** — 現状構成や手順を示す `flowchart TD`。問題の焦点となるノードだけを警告色にすると、図自体が論点を語る。
    - **シーケンス図** — クライアント・サーバー・外部サービスなど、登場者間のやり取りは `sequenceDiagram` で示す。
- 図の数に上限は設けず、1 つの構造につき 1 図とする。構造のない内容（単純な列挙や 1〜2 文で足りる説明）を図にせず、同じ構造の図を焼き直して並べない。同じ内容を図と文章で二重に述べない。
- `flowchart` は `classDef` でノード種別を色分けし、フェンス内 frontmatter の `title:` でラベルを付ける。
- ダイアグラムは flowchart / sequenceDiagram / erDiagram / stateDiagram / pie に限定し、ノードラベルに HTML タグを入れない。

## レンダリング互換性

ADR は `md-preview` でそのまま HTML 化できる形式を保つ。

- ファイル冒頭の frontmatter は書かない（1 項目でも書くと badge / lead / published / updated のすべてが必須になる）。
- Callout は `<Callout type="note|warning|caution">本文</Callout>` と書く。`label="..."` で見出し文字を上書きできる。GitHub 上ではタグが素通しになるが本文は読める。

## チェック

初稿を書き終えたら、まず記述者自身が以下の項目を確認して違反を修正する。その後、記述者とは別の人間またはエージェントが同じ項目で確認し、指摘の修正を経てから提出する。

- [ ] h1 のタイトルが 1 つだけあり、h2 は `背景`・`判断`・`根拠` の 3 つのみである
- [ ] h1 にバッククォート・空白・`/` が含まれていない
- [ ] ファイル名が `YYYYMMDD-hhmm-{h1のタイトル}.md` の形式で、タイトル部分が h1 と完全一致している
- [ ] 判断の結論が冒頭 1〜2 文で読み取れる
- [ ] 本文に frontmatter・日付・状態ラベルがない
- [ ] h1 を除き、コード識別子がすべてインラインコードになっている
- [ ] 地の文の 1 文は 100 字以内、箇条書きの 1 項目は 80 字以内・読点 2 つまでである（疑わしい文は数えて確認する）
- [ ] 同じ事実・同じ数値が複数セクションで繰り返されていない
- [ ] `理由` が背景の再説明になっていない
- [ ] 判断の材料になった数字と名前の具体値が残っている
- [ ] 予定・約束の表現がなく、未決の論点は `未決事項` に整理されている
- [ ] 関係・流れ・収束など構造を持つ説明は mermaid で図示されている（逆に、構造のない内容を図にしていない）
- [ ] mermaid には `title:` があり、flowchart / sequenceDiagram / erDiagram / stateDiagram / pie のみを使っている
